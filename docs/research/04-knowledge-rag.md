# Knowledge / RAG：项目知识、长期记忆与受控上下文注入

## 为什么要做

Agent 不能只依赖模型参数。它需要读取用户项目文件、历史对话、长期偏好和任务相关资料。Knowledge / RAG 的目标，是把外部知识以可控方式注入上下文，让模型回答时既能利用项目材料，又不被材料中的恶意指令接管。

Vela 的知识系统不是单独的搜索产品，而是 Agent 执行链路的一部分。

## 要解决的问题

RAG 要解决三个矛盾：上下文窗口有限，项目知识分散，外部文本不可信。直接把所有文件塞进 prompt 会超限，也容易让文档里的指令污染系统提示。

长期记忆还有额外问题：模型可能把临时信息误写成永久偏好，也可能记住不该记的敏感或指令性内容。因此 Vela 同时做了知识检索和记忆策略。

## 设计目标

Vela 的目标是让知识进入上下文，但不夺走控制权。项目文件通过 chunk 和 embedding 检索，记忆通过重要性、相关性和策略过滤进入上下文，附件文本用不可信标记包裹，retrieved context 作为系统动态段落注入，而不是伪装成用户消息。

这个目标很明确：知识用于增强回答，不用于改写系统规则。

## 核心思想

核心思想是受控检索与受控注入。检索阶段尽量选出相关片段，注入阶段明确告诉模型这些内容是外部资料，不能当成系统指令。长期记忆也不是模型随意写入，而是经过显式确认或后台抽取，再经过策略过滤。

当前源码没有体现独立 rerank 模型。检索主要是 embedding 向量相似度，失败时退化为关键词匹配。这个实现比较朴素，但符合先打通工程链路的阶段。

## 架构设计

Knowledge / RAG 在 Vela 中可以分为四部分：

- 文档索引：后台任务从项目文件中抽取文本，切 chunk，计算 embedding，写入知识表。
- 检索：根据查询 embedding 做向量检索，缺少 embedding 时退化为关键词搜索，并受 token budget 限制。
- 上下文组装：Context Builder 把检索结果注入系统动态段落，避免把它当作用户指令。
- 记忆：显式记忆需要用户确认，自动记忆由后台任务提取，并受 memory policy 约束。

## 为什么这样设计

把 RAG 放在 Context Builder 中，而不是让模型直接调用搜索工具，可以让系统对上下文拥有最终控制权。哪些知识可用、多少 token、如何包裹、是否允许代码执行，都由工程策略决定。

记忆采用显式确认和后台抽取结合，是为了降低模型乱记的风险。显式记忆保证用户意图，后台抽取提高体验，但必须经过 pending、importance、confidence 和 policy 控制。

## 项目体现

源码中的主要证据包括：

- `backend/biz/services/knowledge_service.py` 体现 chunk、embedding、向量搜索、关键词退化和 token budget。
- `backend/workers/tasks/knowledge_tasks.py` 体现项目文件异步索引。
- `backend/biz/infra/embeddings.py` 体现 embedding 依赖系统模型配置，失败时返回空结果。
- `backend/engine/loop/context_builder.py` 体现 retrieved context、memory、附件和工具上下文的统一组装。
- `backend/biz/domain/memory_policy.py` 体现对指令型、策略型、工具型记忆的拒绝。
- `backend/engine/tools/builtins/remember_fact.py` 体现显式记忆需要用户确认。

需要注意：当前注册 API 中知识路由并不是主入口，知识能力主要体现在项目文件和 Agent 上下文链路里。

## Trade-off

朴素 RAG 的优点是简单可靠，缺点是相关性和可解释性有限。没有 rerank 时，embedding 相似度可能选出语义接近但任务不相关的片段；只有关键词退化时，又可能漏掉同义表达。

把检索结果注入系统段落可以降低 prompt injection 风险，但也可能让模型对资料的引用不够自然。严格治理和表达灵活性之间存在取舍。

## 优点

Vela 的优势在于它把 RAG 放进 Agent 运行边界，而不是独立做一个向量搜索 demo。检索结果会经过上下文预算、提示包装、附件处理和 AG-UI retrieved context 事件输出。

记忆系统也不是简单追加文本，而是区分显式记忆、自动抽取、重要性、状态和策略拒绝，这对长期 Agent 很重要。

## 缺点

当前 RAG 能力还比较基础：chunk 策略简单，embedding batch 是尽力式串行，缺少 rerank、混合 BM25、查询改写、引用评估和召回质量监控。知识路由也没有作为当前主 API 暴露。

记忆抽取依赖模型和启发式 fallback，仍可能出现噪声，需要更强的人工治理和过期策略。

## 可以如何优化

可以引入 hybrid retrieval：向量、关键词、结构化元数据和 rerank 结合。对项目代码、文档、聊天历史应采用不同 chunk 策略。还可以增加检索评测集，记录每次 retrieved context 是否被模型实际使用。

记忆方面可以增加生命周期：过期、合并、冲突检测、用户可编辑、项目级与全局级隔离，以及基于任务类型的记忆召回策略。

## 工程经验

RAG 的难点不在向量库，而在上下文边界。检索结果必须被视为不可信输入，尤其是用户上传文件和工具输出。Vela 把资料包进动态段落和不可信标记，是值得保留的工程习惯。

长期记忆不能只追求“多记”。真正有用的是少量高质量、可解释、可删除、不会污染系统策略的记忆。

## 踩坑总结

不要把 embedding 检索当成答案质量保证。没有 rerank 和评测时，召回结果只能算候选上下文。

不要把文件内容直接作为用户消息注入。这样会提高 prompt injection 风险，也会破坏对话历史的语义边界。

## 值得分享的观点

Agent 的知识系统不是“搜索一下再回答”，而是一个上下文治理系统。检索、预算、注入、引用、记忆、拒绝策略缺一不可。

## 一句话总结

Vela 的 Knowledge / RAG 已经形成项目知识和长期记忆的工程链路，但 rerank、评测和知识治理仍有明显演进空间。
