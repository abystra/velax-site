# 整个项目架构：同仓多进程的 AI Agent 平台

## 为什么要做

Vela 的项目结构服务于一个目标：把 Agent 平台拆成可独立演进的业务控制面、模型执行面、共享基础设施和前端体验。它不是一个简单后端加前端项目，而是一个完整 Agent 产品雏形。

这种架构是为复杂度做准备的。只要 Agent 开始调用工具、执行代码、接入知识、写记忆、输出事件和触发后台任务，单进程单模块就很难长期维护。

## 要解决的问题

项目要解决模块边界、依赖方向、运行隔离和扩展性问题。业务 API 不能被模型执行阻塞，Engine 不应该直接处理所有用户和平台治理，Workers 不应该混入同步请求链路，前端也不应该读 Engine 内部状态。

因此架构需要明确谁拥有事实源，谁负责执行，谁负责异步收尾，谁负责协议展示。

## 设计目标

Vela 的架构目标是：控制面和执行面解耦，实时流和持久事件分离，共享模型和配置复用，工具和协议插件化，后台任务异步化，前端通过协议消费运行状态。

这让项目能同时支持 Web、Admin、Desktop App、AG-UI 接入、MCP、Skill、A2A 和沙箱能力。

## 核心思想

核心思想是同仓分层。代码在一个仓库中，但运行时拆成多个进程和职责边界。这样既保留跨模块协作效率，又避免所有能力堆在一个服务里。

Biz 是控制面，Engine 是执行面，Shared 是基础模型与工具层，Workers 是异步治理面，Frontend 是交互面。Docker 和部署配置把这些进程组合起来。

## 架构设计

后端主要分为：

- `backend/biz`：用户、会话、Turn、消息、文件、项目、平台配置、Agent 管理、MCP 管理、Skill 管理、回调和 SSE。
- `backend/engine`：Turn 执行、上下文构建、Agent Loop、模型适配、工具、Skill、Sandbox、AG-UI 事件、A2A handoff。
- `backend/shared`：数据库模型、配置、Nacos、AG-UI 扩展、工具策略、通用 schema。
- `backend/workers`：Celery 后台任务，包括记忆提取、知识索引、标题、审计、usage 和 reaper。

前端包括 `apps/web`、`apps/admin` 和 `apps/app`，并通过共享包复用 Agent 协议、运行时和 UI 组件。

## 为什么这样设计

Biz 和 Engine 分离，是因为业务系统需要稳定处理认证、权限、事件和平台治理，而 Engine 运行模型与工具调用，负载和失败模式完全不同。把它们拆开后，Engine 可以被容量控制、租约和 reaper 管理，Biz 可以保持业务入口稳定。

Workers 独立，是因为很多事情不应该阻塞用户 Turn，例如记忆抽取、标题生成、知识索引和审计。Shared 存在，是为了避免 Biz 和 Engine 各自定义一套状态、配置和协议模型。

## 项目体现

当前实现基线和源码体现出明确运行链路：

1. Biz 接收用户消息，创建 Turn，写 durable event。
2. Biz 调用 Engine 内部启动接口。
3. Engine 抢占 Turn，构建工具、Skill、上下文和 Agent Loop。
4. Engine 通过 EventBus 同时写 Redis Stream 和 durable events。
5. Biz SSE 把 AG-UI 事件推给前端。
6. Engine 完成后回调 Biz。
7. Biz 触发 Workers 做记忆、标题、审计和 usage 等后处理。

这一链路体现了控制面、执行面、事件面和异步面的分离。

## Trade-off

这种架构比单体复杂。开发者需要理解多个服务、数据库事件、Redis Stream、Celery、内部回调和前端协议。一次 Turn 的问题可能跨越多个模块。

但它换来的是更强的扩展性和故障隔离。Engine 压力大时可以独立扩容，Workers 慢时不阻塞主请求，前端恢复依赖 durable events 而不是 Engine 内存。

## 优点

项目模块边界比较清晰。业务、执行、共享、后台和前端各自承担不同职责。工具、协议、沙箱、Nacos、Skill、A2A 都被放在可扩展位置，而不是散落在业务 API 中。

事件事实源的设计也增强了恢复能力。Redis Stream 只是实时缓冲，durable events 才是长期事实，这让断线重连和历史投影成为可能。

## 缺点

复杂度已经不低。内部 API 安全、Turn 租约、状态迁移、事件一致性、工具幂等、Worker 重试和前端投影都需要持续测试。文档和实现基线也必须保持同步，否则团队很容易误判当前能力。

前端实际存在 Desktop App，但部分部署配置主要覆盖 Web/Admin，产品形态和部署形态还需要进一步统一。

## 可以如何优化

可以增加架构一致性测试，例如 Turn 从提交到完成的端到端事件断言，AG-UI 历史恢复与实时流一致性测试，工具执行幂等测试，租约过期和 reaper 测试。

还可以将 provider 状态、模型路由、工具策略、沙箱健康、Nacos 同步和 Workers 延迟统一到 Admin 可观测页面。

## 工程经验

Agent 平台的架构设计要从失败模式倒推。模型失败、工具失败、沙箱失败、前端断线、Engine 崩溃、Worker 延迟，都要有对应边界。

Vela 的一个重要经验是不要把实时流当事实源。实时层用于体验，持久层用于恢复和审计。

## 踩坑总结

不要让 Engine 直接承担业务权限和用户入口，否则执行失败会污染控制面。不要让 Biz 直接运行工具和模型，否则业务服务会被不确定执行拖垮。

不要忽略后台任务的治理。记忆、知识、审计和 usage 虽然是异步，但它们会影响长期体验和平台可信度。

## 值得分享的观点

一个 Agent 项目从 demo 走向平台，关键不是多接几个模型，而是把控制面、执行面、协议面和治理面拆开。

## 一句话总结

Vela 的整体架构是同仓多进程的 Agent 平台设计，用 Biz/Engine/Shared/Workers/Frontend 分离不确定执行和确定业务治理。
