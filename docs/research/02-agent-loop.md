# Agent Loop：把不确定模型调用变成可恢复的执行系统

## 为什么要做

Agent Loop 是 Vela 的执行核心。普通聊天系统只需要一次模型调用，而 Agent 需要在模型、工具、观察、再推理、暂停、恢复和终止之间循环。这个循环如果没有工程约束，很容易出现无限调用、重复工具、上下文膨胀、取消无效、工具副作用重复执行等问题。

Vela 做 Agent Loop 的目的，是把模型的开放式行为压进一个有状态、有预算、有停止条件、有事件记录的运行系统。

## 要解决的问题

Loop 要同时处理几类问题：模型输出不可预测，工具调用可能失败或重复，用户可能中断，工具可能要求审批，沙箱可能超时，模型 provider 可能在流式输出前失败，长上下文可能超限。

这些都不是普通 Workflow 的线性节点能轻易覆盖的。Agent Loop 必须允许“模型决定下一步”，同时又要保证系统能停止、能恢复、能审计。

## 设计目标

Vela 的 Loop 目标可以概括为：每一步都可观察，每个副作用都可约束，每种终态都可落库，每次恢复都有上下文依据。

它不是为了追求形式上的自动推理，而是为了在工程上解决：什么时候继续，什么时候停，什么时候让用户介入，什么时候降级，什么时候把工具结果重新交给模型。

## 核心思想

Loop 的基本思想是 Reasoning 与 Acting 交替。模型先基于上下文生成内容和工具调用，系统执行工具并观察结果，再把观察结果加入下一轮上下文。循环直到模型不再请求工具、触发人工输入、被取消、失败预算耗尽或达到最大迭代次数。

Reflection 在当前源码里不是一个独立“反思 Agent”。它更多体现为工程化反馈机制：失败预算、重复工具预算、最终合成、工具结果回灌、文档产物校验、记忆提取和中断恢复。这种做法更务实，也更可控。

## 架构设计

Vela 的 Loop 包含这些关键环节：

- Planning：模型基于系统提示、会话历史、知识、记忆、工具摘要形成下一步动作。
- Tool Call：模型输出工具调用，Registry 校验 allowlist、权限、风险、调用次数和审批策略。
- Observe：工具结果被包装成受限输出，重新进入下一轮上下文。
- State：Turn 状态在 pending、building、calling、streaming、executing、waiting_input、completed、failed、cancelled、timeout 间流转。
- Context：首轮构建完整上下文，后续增量追加 assistant 和 tool 消息，避免反复重建。
- Memory：显式记忆需要用户确认，自动记忆由后台任务在完成后提取。
- Stop Condition：最大迭代、失败预算、重复预算、取消、超时、等待用户输入、正常完成。

## 为什么这样设计

普通 Workflow 的节点是预先定义的，而 Agent 的下一步经常由模型在运行时决定。Vela 采用 Loop，是因为它要支持模型在任务过程中动态选择工具、读取观察结果并继续推理。

但 Vela 没有把控制权完全交给模型。工具 allowlist、审批、风险等级、调用次数、失败预算、取消信号、租约和状态机都在外部约束模型行为。这是 Agent Loop 与“模型自由发挥”的关键区别。

## 项目体现

源码体现集中在这些模块：

- `backend/engine/loop/agent_loop.py` 承载模型流、工具循环、取消、失败预算、重复预算、最终合成和中断。
- `backend/engine/loop/context_builder.py` 从 durable events、快照、知识、记忆、附件和工具策略中构建上下文。
- `backend/engine/loop/compact.py` 在上下文接近上限时按交换边界压缩，避免切断工具调用和工具结果。
- `backend/biz/domain/turn_state.py` 定义 Turn 生命周期，说明 Loop 不是靠内存状态自说自话。
- `backend/engine/runner/journal.py` 和租约相关模块体现执行步骤的幂等与恢复意识。

## Trade-off

Loop 的优势是灵活，代价是难以预测。模型可能多走几轮，工具可能被重复调用，成本和延迟都不如固定 Workflow 稳定。

Vela 的取舍是允许动态循环，但用预算和状态机限制失控。它没有追求“完美自主”，而是优先让失败可见、可停、可恢复。

## 优点

这种 Loop 能处理真实 Agent 任务中的非线性行为。例如工具执行后还需要继续推理，用户审批后才能恢复，模型流式输出前失败可以切换候选模型，取消信号可以关闭上游流以减少计费。

它也让前端体验更稳定。AG-UI 事件能展示 reasoning、tool call、tool result、状态增量和中断卡片，而不是只有最终文本。

## 缺点

Loop 的复杂度很高。状态、事件、工具结果、模型消息和上下文压缩之间存在强约束，一旦某个环节处理不好，就会出现重复执行、上下文丢失、终态不一致或取消无效。

当前 Reflection 主要是工程化反馈，并非完整的自我评估与计划修正系统。如果任务需要强规划和多阶段验证，还需要更显式的反思与评估模块。

## 可以如何优化

可以增加更结构化的 Reflection：在工具失败、循环接近上限、产物生成后自动做计划修订和质量检查。也可以引入更细的 stop policy，例如按工具风险、模型成本、用户等级和任务类型动态调整迭代预算。

另外，Loop 与未来 Graph Runtime 可以结合：常规任务走 Loop，强流程任务走显式图，复杂任务由 Planner 选择执行策略。

## 工程经验

Agent Loop 必须从第一天就考虑取消、重试、幂等和终态。只写“模型调用工具再继续”的 demo 很容易，但一旦接入用户界面、后台任务、沙箱和远端工具，Loop 会变成分布式系统问题。

Vela 的经验是把 Loop 输出为事件，而不是只在内存里维护状态。事件流让前端、恢复、审计和后台处理都有统一依据。

## 踩坑总结

最大的坑是工具副作用重复执行。Vela 用 step journal、idempotency key、fencing token 和 max_calls_per_turn 降低风险，但这类问题仍然需要持续治理。

另一个坑是取消。取消不能只改数据库状态，还要通知正在运行的 Engine，并尽量关闭模型流和沙箱执行，否则用户看到取消了，后台仍在消耗成本。

## 值得分享的观点

Agent Loop 的本质不是“自动思考”，而是把开放式模型行为纳入有限状态机、事件日志和预算系统。真正的智能体验来自动态循环，真正的工程质量来自边界约束。

## 一句话总结

Vela 的 Agent Loop 用状态、事件、预算和工具策略，把不可预测的模型推理变成可恢复的工程执行过程。
