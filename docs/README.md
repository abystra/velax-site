# Vela 技术思想研究知识库

## 为什么要做

Vela 不是一个只有聊天接口的项目，而是一个把 Agent 运行时、业务系统、协议层、工具系统、沙箱、知识库和平台治理放在一起考虑的工程项目。这个知识库的目的，是把源码背后的设计动机提炼出来，持续沉淀成技术博客、Podcast 和技术分享素材。

它不做 API 文档，也不按类和函数解释代码。每篇文章都围绕一个问题：作者为什么这样设计，以及这种设计解决了什么工程矛盾。

## 要解决的问题

这套文档主要解决三个问题：

- 把 Vela 中分散在 `backend/biz`、`backend/engine`、`backend/shared`、`backend/workers`、`frontend` 和部署目录里的设计意图串起来。
- 区分“源码已经实现的工程事实”和“未来可以演进的架构愿景”。
- 为后续持续写作提供稳定主题，而不是每次从零阅读源码。

## 设计目标

知识库采用主题化组织，每个主题都固定回答为什么、目标、思想、架构、项目体现、Trade-off、优缺点、优化方向和经验总结。这样可以让内容既适合写长文，也适合拆成分享提纲或播客脚本。

## 核心思想

Vela 的核心设计不是“把大模型接进系统”，而是把不确定的模型行为包进可恢复、可审计、可治理的工程边界里。源码反复体现了几条主线：业务面和执行面分离，事件日志作为事实源，工具和协议插件化，代码执行必须进入沙箱，知识与记忆必须经过策略约束。

## 为什么这样设计

这些主题不是按目录拆出来的，而是按 Agent 平台最容易出问题的工程边界拆出来的：谁负责状态，谁负责执行，谁负责协议，谁负责工具，谁负责隔离，谁负责配置和发现。这样更适合持续输出设计思想，而不是停留在源码导读。

## 项目体现

主要证据来自 Vela 源码与当前实现基线：

- `backend/biz`：业务 API、会话、Turn、事件、用户与平台治理。
- `backend/engine`：Agent Loop、上下文构建、工具执行、协议事件、沙箱、Skill、MCP、A2A handoff。
- `backend/shared`：配置、数据库模型、协议扩展、Nacos 适配、共享策略。
- `backend/workers`：异步任务、记忆提取、知识索引、Turn reaper。
- `frontend`：Web、Admin、Desktop App 以及 Agent 协议相关包。
- `方案/技术栈/15-当前代码实现基线.md`：当前运行边界和已知风险的基线说明。

## Trade-off

这套知识库会优先记录源码能支撑的结论。对于源码没有完整闭环的方向，例如通用 Graph Runtime、完整 Supervisor/Worker 多智能体、Rerank、Docker Provider、Nacos 配置热更新，会明确标注为未完整体现，而不是把未来愿景写成当前事实。

## 优点

主题化之后，Vela 的复杂度被拆成了可以反复讨论的工程问题。每个主题既能独立成文，也能组合成完整的 Agent 平台架构叙事。

## 缺点

这种写法会牺牲一部分源码细节，不适合作为开发者 API 手册。如果未来源码变化较大，文档需要跟着重新校验项目体现部分。

## 可以如何优化

后续可以把每篇文章继续扩展成三种形态：技术博客版、分享大纲版、Podcast 口播版。同时可以增加源码证据索引，把关键模块和设计观点做成双向映射。

## 工程经验

分析这类 Agent 项目时，不能只看模型调用和工具列表。真正有价值的是运行边界：状态如何恢复、事件如何持久化、工具如何审批、沙箱如何隔离、协议如何演进、失败如何收敛。

## 踩坑总结

最容易踩的坑是把概念名词当成实现事实。例如看到 A2A 就认为已经有完整多智能体运行时，看到 Workflow 就认为有通用图执行器，看到 Nacos 就认为所有配置都支持热更新。本文档会尽量避免这种误判。

## 值得分享的观点

Vela 最值得分享的不是某个单点功能，而是它尝试回答了 Agent 工程化的核心问题：如何让一个会调用工具、会暂停、会恢复、会产生副作用的模型运行过程，进入可治理的软件系统。

## 一句话总结

这是一套面向持续写作的 Vela 架构思想知识库，重点记录设计理由、工程取舍和未来演进，而不是解释代码。
